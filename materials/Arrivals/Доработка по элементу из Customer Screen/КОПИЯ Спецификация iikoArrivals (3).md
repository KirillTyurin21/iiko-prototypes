# Спецификация iikoArrivals

1. ## Введение

   1. ### Цель

Целью данного документа является описание задачи разработки сервиса iikoArrivals для отображения клиентских заказов на табло в заведении общественного питания \[далее \- Заведение\]. Сервис будет получать информацию о заказах от системы iikoFront и выводить их на табло для общего просмотра клиентами и персоналом.

2. ### Система iikoArrivals

Система iikoArrivals представляет собой веб\-приложение, которое предназначено для отображения электронной очереди в Заведениях. Функциональные возможности данной системы включают в себя отображение информации о заказах, таких как номер заказа, статус заказа, время приготовления/ожидания, другие данные,  а также уведомления о готовности заказов (в том числе и звуковые оповещения),  смену изображений на табло (в рекламных целях).

3. ### Настройки

Для администрирования и настройки отображения информации на экране электронной очереди, предусмотрена возможность выбора нескольких областей для показа информации и применения к ним различных настроек, включая включая типы заказов, номера столов, элементы заказов, статусы заказов и формата отображения статуса заказов.

2. ## Описание задачи

Сервис iikoArrivals представляет собой систему электронной очереди для Заведений, обеспечивающую отображение информации о заказах на экране. Основной целью разработки является создание удобной и эффективной системы для клиентов, курьеров и персонала ресторана, позволяющей наглядно отслеживать текущее состояние заказов.

Сервис должен поддерживать различные типы заказов, включая обычные заказы, самовывоз заказа клиентом и доставку курьером. Каждый заказ может находиться в различных статусах, таких как "Готово", "Готовится", "Подано", что позволяет как клиентам, так и курьерам оперативно управлять заказами и отслеживать их статус.

Основные задачи разработки включают получение информации о заказах от кассовой системы, хранение данных о настройках и конфигурации отображения заказов на экране, а также отображение заказов на экране с учетом текущего состояния заказа. Для обеспечения удобства использования, сервис должен иметь интуитивно понятный интерфейс как для для администраторов так и для пользователей системы.

Целью разработки является создание надежного, быстродействующего и легко масштабируемого сервиса, который будет удовлетворять потребности как клиентов, так и персонала ресторана, обеспечивая эффективное управление заказами и повышая общую удовлетворенность клиентов.

3. ## Функциональные требования

   1. ### Роли пользователей

В системе предусмотрены три типа ролей пользователей:

Администратор:

* Этот уровень доступа предоставляется сотрудникам компании iiko.  
* Имеет полные права доступа ко всем функциям и возможностям системы.  
* Может создавать, редактировать и удалять контролы.  
* Может создавать, редактировать и удалять любые настройки для экранов.

Партнер:

* Этот уровень доступа предоставляется сотрудникам партнерских компаний.  
* Может иметь ограниченные права доступа в зависимости от договоренностей с компанией iiko.  
* Может создавать, редактировать и удалять только те контролы, которые были созданные с привязкой к учетной записи с ролью Партнер контролы.  
* Может создавать, редактировать и удалять только те настройки экранов, которые были созданные с привязкой к учетной записи с ролью Партнер.

Обычный пользователь:

* Этот уровень доступа предназначен для пользователей, которым не требуется административный доступ.  
* Обычные пользователи могут просматривать информацию, но не имеют права создавать, редактировать или удалять контролы и изменять настройки тем, только применять готовы настройки.

  2. ### Управление контролами

Этот раздел панели администрирования, должен предоставлять возможность администраторам создавать, редактировать и удалять контролы, которые отображаются информацию о заказа на экране. Контролы представляют собой набор элементов и констант, используемых для отображения различных аспектов заказов, таких как номера заказа, статус заказа и другие.

Для выполнения операций с контролами необходимо, в панель администрирования в боковую панель добавить новый раздел "Дисплей Arrivals". Этот раздел будет содержать подраздел "Контролы", где будут располагаться все контролы связанные с отображением информации о заказах на экране Заведения. Подраздел контролы, должен быть доступен только пользователям с правами администратора.

2. ### Просмотр перечня контролов

Администратор может просматривать перечень всех созданных контролов. Просмотр перечня контролов, должен осуществляться на вкладке “Контролы”.

*Страница перечня контролов*

Страница перечня контролов состоит из:

* Названия \- Название, данное контролу для его идентификации.  
* Создан \- Этот параметр указывает, кто создал данный контрол. Он может принимать значения, такие как "iiko", если контрол был создан Администратором, или "Моя", если контрол создан партнером.

На странице перечня контролов должна быть предусмотрена функция выбора контрола, которая позволяет пользователю с выделенными правами \[далее \- Пользователю\] просматривать внешний вид контрола на экране предварительного просмотра. Пользователю должна быть доступна возможность изменять возможные статусы контрола на этом экране, чтобы наглядно оценить, как выглядит контрол в различных статусах. Выбор статуса, представляет собой выпадающий список всех статусов, которые доступны для определения контролом.

3. ### Создание нового контрола

Администратор:

* Может создать новый контрол двумя способами:  
  * Нажав на кнопку “+” в верхнем правом углу экрана.  
  * Нажав на кнопку “Редактировать” у любого контрола, в графе “Создан” которого указано значение “iiko”. При переименовании такого контрола, должен быть создан новый контрол.

Партнер:

* Имеет право создавать только свои контролы.

Обычный пользователь:

* Не имеет права создавать контролы.

  4. ### Редактирование контрола

Администратор:

* Может редактировать контролы, нажав на кнопку “Редактировать” у любого контрола, на вкладке “Контролы”. Контролы, в графе “Создан” которых, указано значение “iiko”, могут быть отредактированы. При редактировании создается только копия контрола.

Партнер:

* Имеет право редактировать только свои контролы.   
* Могут клонировать существующий контрол для дальнейшего редактирования, однако клонирование происходит уже как собственный элемент.

Обычный пользователь:

* Не имеет права редактировать контролы.

  5. ### Удаление контрола

Администратор:

* Может удалить любой контрол, в графе ”Создан” которого указано любое значение.  
* Для удаления контрола, необходимо на вкладке “Контролы” нажать на кнопку “Корзина”.  
* После нажатия, должно отобразиться диалоговое окно с запросом на подтверждение удаления контрола.  
* После подтверждения, контрол должен удалиться из системы и перестать отображаться в перечне контролов на вкладке “Контролы”.

Партнер:

* Имеет право удалять только свои контролы.

Обычный пользователь:

* Не имеет права удалять контролы.

  3. ### Страница создания и редактирования контрола

На данной странице Пользователь имеет возможность создать новый контрол или отредактировать существующий. Эта страница представляет собой интуитивно понятный интерфейс, который обеспечивает гибкую настройку внешнего вида и функциональных характеристик контрола.  
![][image4]  
*Интерфейс экрана создания контрола. Красная рамка \- область контрола.*  
Компоненты страницы:

* Название \- указывается название контрола, которое будет отображаться в перечне контролов.  
* Статус \- это параметр контрола, определяющий внешний вид контрола в рабочей области экрана в зависимости от выбранного статуса. Для каждого статуса должны настраиваться отдельные элементы данных и их визуальный вид, что позволяет управлять поведением и внешним видом контрола в зависимости от его статуса.  
* Показать границы \- компонент отображает общую границу контрола вместе со всеми его элементами. Позволяет Пользователю идеть физические границы контрола на экране и убедиться, что его расположение и размеры соответствуют заданным параметрам.  
* Добавить элемент \- кнопка "Добавить элемент" позволяет Пользователю добавить новый элемент для отображения в рабочей области. Нажатие на эту кнопку открывает меню с доступными опциями для выбора элемента данных.  
* Очистить элементы \- Кнопка "Очистить элементы" удаляет все добавленные элементы из рабочей области.  
* Область контрол \- определяет общие настройки внешнего вида контрола.

При создании нового контрола автоматически создается область контрола, которую нельзя удалить.  
Область контрол размещается в рабочей зоне, которая охватывает максимально допустимые размеры согласно разрешению экрана, а элементы должны быть добавлены внутрь области контрола.  
Данная область будет служить контейнером для размещения элементов данных таких как, номер заказа, номер стола и т.п. и представляет собой прямоугольную область на экране.  
Если элементы выходят за границы области, они могут быть не видны на экране, поэтому область контрола помогает правильно организовать и позиционировать элементы на экране.

 Ниже указаны все доступные элементы и их настройки:

1. #### Номер заказа

В качестве элемента данных, в файле конфигурации config.js в Arrivals использовался параметр order.number.  
В IOrder Interface, указан как свойство Number  
В IDeliveryOrder Interface, указан как свойство Number  
Настройки элемента: 

* Название: Минимальная длина номера заказа  
  * Тип: Целое положительное число  
  * Описание: Определяет минимальную длину номера заказа при выводе.  
  * Пример: Если установить значение 3, то номер заказа будет заполняться символами слева до достижения требуемой длины.  
  * Значение по умолчанию: 1  
* Название: Символ заполнения  
  * Тип: Строка  
  * Описание: Определяет значение, которое будет заполнять номер заказа в случае, если его длина меньше заданной в параметре "Минимальная длина номера заказа".  
  * Пример: Если установить символ "Р", а минимальная длина номера заказа равна 5, то если номер заказа будет "123", то он будет отображен как "РР123". Можно указать только один символ.  
  * Значение по умолчанию: “0”

Настройки источников заказа:

* Название: Префикс источника заказа  
  * Тип: Строка  
  * Описание: Определяет символ или несколько символов, которыу будут приписываться к номеру заказа для обозначения его типа источника. Для каждого источника необходимо указать свой набор символов или один символ. Если символы не указаны для источника, то такой источник не выделяется в отображении.  
* Название: Маска источника заказа.  
  * Тип: Строка  
  * Описание: Определяет маску, для префикса источника заказа. Должна быть возможность обозначить Маску. Где X \- Буква, N \- номер заказа.  
* Пример:  
  * Маска: XN  
    * Номер заказа: 15  
    * Тип заказа: Яндекс.ЕДА \- YE  
    * Отображение: YE15  
  * Маска: N-X  
    * Номер заказа: 15  
    * Тип заказа: Delivery \- D  
    * Отображение: 15-D  
* Название: Минимальная длина номера заказа  
  * Описание: Только для источника заказаю  
* Название: Символ заполнения  
  * Описание: Только для источника заказа.

Настройки источника заказа, имеют приоритет выше чем у настроек контрола.

2. #### Показ столов

В качестве элемента данных, в файле конфигурации config.js в Arrivals использовался параметр order.tableNum.  
В IOrder Interface, номер заказа указан как свойство Tables  
В IDeliveryOrder Interface, указан как свойство Tables  
Настройки элемента:

* Название: Показ столов  
  * Тип: целые положительные числа, соответствующие номерам столов с системе iikoFront  
  * Описание: Определяет выводимые или скрываемые номера столов, для которых отображаются заказы на экране. Номера столов указываются через запятую, а промежутки столов через тире.  
  * Пример: Если установить значения "1, 3-5, 7", будут отображены заказы для столов с номерами 1, 3, 4, 5, 7  
* Название: Тип значения  
  * Тип:  Строка  
  * Описание: Определяет, будут ли отображаться столы на экране. Значение "Показать столы" указывает, что значения параметра “Показ столов” определяет столы которые будут отображены на экране заказов \- остальные скрыты, а "Скрыть столы" \-  указывает, что значения параметра “Показ столов” определяет столы которые будут скрыты на экране заказов \- остальные отображены.  
  * Пример: Если установить значение “Скрыть столы”, то на экране заказов будут выводиться заказы только для столов 1, 3, 4, 5, 7\. Заказы для остальных столов, будут скрыты из вывода на экране.  
  * Значение по умолчанию: null \- Выводить все столы.

    3. #### Текущий статус заказа

В качестве элемента данных, в файле конфигурации config.js в Arrivals использовался параметр order.status.  
В IOrder Interface, номер заказа указан как свойство Status  
В IDeliveryOrder Interface, указан как свойство Status  
Настройки для данного элемента не определены.  
Пример вывода: Выводит текущий статус заказа, если заказ приготовлен, то выведется значение ”Готов”.

4. #### Время начала приготовления заказа

В качестве элемента данных, в файле конфигурации config.js в Arrivals использовался параметр order.cookingStart  
В IOrder Interface, номер заказа указан как свойство BillTime  
В IDeliveryOrder Interface, указан как свойство BillTime  
Настройки для данного элемента:

* Название: Настройка вывода значения  
  * Тип: Маска вывода даты  
  * Описание: Определяет маску, используемую для отображения даты и времени. В маске могут использоваться следующие обозначения: YYYY \- год, MM \- месяц, dd \- день, HH-часы, mm \- Минуты, Sec \- секунды.  
  * Пример: Если установить маску “HH:mm”, то будет отображено время в формате  то время будет отображаться в формате "часы:минуты", например 12:01.  
  * Значение по умолчанию: HH:mm

    5. #### Время завершения приготовления заказа

В качестве элемента данных, в файле конфигурации config.js в Arrivals использовался параметр order.cookingFinish  
В IDeliveryOrder Interface, указан как свойство CookingFinishTime   
Настройки для данного элемента:

* Название: Настройка вывода значения

3. ### Настройки макета

Настройки макета предоставляют возможность настроить внешний вид элемента заказа с помощью различных параметров. При выборе опции “Макет”, открывается список доступных параметров, которые могут быть изменены в соответствии с потребностями пользователя. Доступные для изменения параметры “Макета”:

* Цвет фона элемента \- Позволяет выбрать цвет фона для элемента заказа с помощью палитры цветов или указать его код (HEX RGB).--  
* Отступы \- задает расстояние от границы рамки экрана элемента до содержимого элемента.(целочисленное значение)  
* Сверху \- Абсолютное расстояние от верхнего края рабочей области.(целочисленное значение)  
* Слева \- Абсолютное расстояние от левого края рабочей области.(целочисленное значение)  
* Ширина \- Ширина контрола.(целочисленное значение)  
* Высота \- Высота контрола.(целочисленное значение)  
* Слой \- Определяет порядок наложения контролов при их перекрытии(целочисленное значение). Чем больше число, тем выше слой.  
* Угол поворота \- Позволяет вращать элемент заказа на определенный угол относительно его центра.(числовое значение). Положительное значение осуществляет поворот по часовой стрелке, отрицательное \- против часовой стрелки.

  4. ### Настройки границ

Настройки “Границ” позволяют определить внешний контур элемента заказа. При выборе опции "Границы", доступны следующие параметры для настройки:

* Ширина \- Позволяет выбрать толщину границы.(целочисленное значение)  
* Цвет \- Позволяет выбрать цвет границы для элемента заказа с помощью палитры цветов или указать его код (HEX RGB).  
* Тип \- Позволяет выбрать стиль границы. Выпадающий список.   
  * Скрытый  
  * Точечный  
  * Пунктирный  
  * Сплошной  
  * Двойной  
  * С выемкой  
  * Рамчатый  
  * Впуклый  
  * Выпуклый  
  * По умолчанию: Скрытый.  
* Скругление углов \- Определяет степень скругления углов границы элемента.(целочисленное значение)

  5. ### Настройки шрифта

Настройки шрифта позволяют определить внешний вид текста элемента заказа. При выборе этой опции открываются параметры, позволяющие настроить различные аспекты шрифта:

* Цвет \- Позволяет выбрать цвет шрифта для элемента заказа с помощью палитры цветов или указать его код (HEX RGB).  
* Размер \- Предоставляет возможность задать размер текста в пикселях.(целочисленное значение)  
* Минимальный размер \- Указывает минимальный размер шрифта. Размер шрифта уменьшается до указанное значение, при невозможности уместить текст в заданных рамках области элемента.  
* Шрифт \- Позволяет выбрать шрифт текста из предустановленного списка доступных шрифтов. Выпадающий список.  
* Выравнивание текста \- Определяет, как текст будет выравниваться внутри границ элемента заказа. Выпадающий список. Варианты:  
  * По левому краю;  
  * По правому краю;  
  * По центру;  
  * По ширине.  
  * По умолчанию: По центру.  
* Стиль шрифта \-  Позволяет выбрать стиль текста. Выпадающий список. Варианты:  
  * normal;  
  * italic;  
  * oblique.  
* Начертание шрифта \- Определяет, будет ли текст написан обычным, подчеркнутым, зачеркнутым  и другими начертаниями. Выбирается из предложенных базовых вариантов выпадающего списка.

* Угол поворота \- Позволяет вращать область контрола на определенный угол относительно его центра.(числовое значение). Положительное значение осуществляет поворот по часовой стрелке, отрицательное \- против часовой стрелки.

Настройки “Границ” позволяют определить внешний контур элемента заказа. При выборе опции "Границы", доступны следующие параметры для настройки:

* Ширина \- Позволяет выбрать толщину границы.(целочисленное значение)  
* Цвет \- Позволяет выбрать цвет границы для области контрола с помощью палитры цветов или указать его код (HEX RGB).  
* Тип \- Позволяет выбрать стиль границы для области контрола. Выпадающий список.   
  * Скрытый  
  * Точечный  
  * Пунктирный  
  * Сплошной  
  * Двойной  
  * С выемкой  
  * Рамчатый  
  * Впуклый  
  * Выпуклый  
  * По умолчанию: Скрытый.  
* Скругление углов \- Определяет степень скругления углов границы элемента.(целочисленное значение)

7. ### Изменение типов статусов заказа контрола

При изменении статуса заказа контрола, система должна дать Пользователю возможность использовать ранее заданные параметры настроек внешнего вида и элементов для нового статуса. Первоначальное переключение на новый статус заказа, должно осуществляться с клонированием всех стилистических настроек и элементов из прежнего статуса заказа контрола и их применения к новому статусу. Это позволит сохранить консистентность внешнего вида контрола при переходе между различными статусами заказа.

После первичного переключения, Пользователь может производить дополнительные изменения в новом виде контрола, чтобы адаптировать его к новому статусу заказа. Также предоставляется возможность очистить элементы и начать компоновку контрола с нуля для выбранного статуса, путем нажатия на кнопку “Очистить элементы”.

Повторное переключение на статус, который уже был ранее выбран и указан, не должно приводить к клонированию и сбрасыванию настроек статуса. Вместо этого система должна сохранять ранее установленные настройки для данного статуса, чтобы обеспечить сохранение информации по внешнему виду контрола для указанного статуса и предотвратить потерю данных и времени при настройке контрола. При необходимости восстановления начального состояния элемента или возврата к предыдущему виду контрола, администратор должен иметь возможность клонировать внешний вид с любого другого статуса. 

![][image11]  
*Перечень статусов для выбора.*

Пример 1:

* Текущий статус заказа \- "Готовится".  
* Пользователь изменяет статус заказа на "Готов".  
* Система использует ранее установленные настройки для предыдущего статуса “Готовится”.  
* Все стилистические настройки и элементы из текущего вида контрола автоматически клонируются и применяются к новому статусу "Готов".

Пример 2:

* Администратор переключает статус обратно на "Готовится", система сохраняет все ранее установленные настройки для статуса “Готово”.  
* Новые настройки не создаются для статуса “Готовится”, и ранее внесенные изменения сохраняются для статуса “Готово”, чтобы обеспечить корректное отображение заранее заданных сведений.

  8. ### Создание новых статусов заказов[^1]

Пользователи, имеют возможность расширять функциональность системы, добавляя новые статусы заказов прямо из панели настроек контрола. Этот процесс позволяет адаптировать систему к потребностям Заведения. Вот более подробное описание этого процесса:

* Пользователь переходит к разделу "Добавить статус" в настройках контрола.  
* В данном разделе предоставляется кнопка “+”, на которую пользователь нажимает для добавления нового поля статуса.  
* Пользователю предлагается заполнить название нового статуса. У пользователя должна быть возможность определить очередность появления нового статуса, путем перетаскивания поля с названием нового статуса левой кнопкой мыши.  
* Для подтверждения и сохранения ввода данных,  пользователь подтверждает создание нового статуса, нажав кнопку "Сохранить".  
* Пользователь, за одну процедуру не должен быть ограничен созданием одного нового статуса.  
* Система проводит проверку на корректность введенных данных и подтверждает успешное создание нового статуса, после чего он автоматически добавляется в список доступных статусов заказов.

9. ### Сохранение контрола

После завершения настройки всех параметров контрола, Пользователь должен указать “Название контрола” и сохранить внесенные изменения, чтобы они вступили в силу. Сохранение контрола осуществляется путем нажатия на кнопку "Сохранить".

2. ### Настройки области

   1. ### Настройка режима работы области

Настройка области осуществляется, через выбор области в рабочем пространстве темы либо через выбор области в перечне добавленных контролов к теме.  
Должны поддерживаться два вида режимов работы области:

* Динамически заполняемая область;  
* Лист.

  1. ### Режим Динамически заполняемой области

Контролы, содержащие информацию о заказе, размещенные в области с типом “Динамически заполняемая область”, отображают заказы по следующему алгоритму:

* Позиция контрола, установленная пользователем внутри указанной области, считается стартовой.  
* Вычисляется параметр margin указанного контрола, отступ от верхнего и левого края границ контролов до границы указанной области.  
* Проверяется доступное место по правому краю от контрола, где свободное место должно быть больше, чем сумма margin слева и ширины контрола.  
* Если размер свободного места больше суммы длины margin и ширины контрола, то следующий контрол заказа, при отображении его на экране заказов, размещается в измеряемой области.  
* Если размер свободного места меньше указанного значения, то следующий заказ размещается ниже предыдущего контрола на следующей строчке, слева-направо, на расстоянии, равном размерности контрола margin сверху.

2. ### Режим области Лист

Контролы, размещенные в области с режимом “Лист”, отображают заказы по следующему алгоритму:

* При размещении контрола, содержащим информацию о заказе, в указанной области, проверяются настройки области Лист по параметру “Положение начала списка”: “От верхнего края” или “От нижнего края”. По умолчанию \- От верхнего края.  
  * Если установлена настройка \- “От верхнего края”, то контрол содержащий информацию о заказе, автоматически размещается в области Лист с параметрами:  
    * margin сверху равным 0;  
    * В правой части контрола устанавливается параметр margin-справа равный значению margin-слева;  
    * Ширина контрола устанавливается значению, равному ширина области Лист минус сумма длин margin-слева и margin-справа;  
    * Отсчет заказов начинается сверху вниз.  
  * Если установлена настройка \- “От нижнего края”, то контрол содержащий информацию о заказе, закрепляется за нижней частью области, с идентичным значением margin.  
* Проверяется доступное место в зависимости от настроек:  
  * Если установлена настройка “От верхнего края”, то проверяется свободно ли место под крайним нижнем контролом до нижнего края границы области.  
    * Если размер свободного места больше Высоты контрола, то следующий контрол заказа, при отображении его на экране заказов, размещается в измеряемой области.  
    * Если размер свободного места меньше указанного значения, не размещается в измеряемой области.  
  * Если установлена настройка “От нижнего края”, то проверяется свободно ли место над крайним нижнем контролом до верхнего края границы области.

## Приложение 1\. Справочник источников

Справочник Источников заказа позволяет администраторам и пользователям управлять списком источников заказов, которые используются на всех кассах и в областях тем Arrivals, внутри одной организации.

Справочник Источников заказа доступен на кассах под кнопкой "Настройки", под блоком “Статусы”. Для доступа к справочнику:

* На интерфейсе перечня Касс нажмите на кнопку "Настройки".  
* В открывшемся боковом окне, под заголовком “Источники заказов”, должен отображаться весь перечень текущих источников.

После доступа к справочнику пользователю будет представлен полный список текущих источников заказа. Каждый элемент списка включает:

* Название источника заказа.  
* Иконка редактирования (карандаш): позволяет переименовать источник.  
* Иконка удаления (корзина): позволяет удалить источник.

Все изменения в справочнике Источников заказа (и в настройках областей) сохраняются в базе данных и моментально отражаются на всех кассах и в областях внутри организации. 

## Управление Источниками заказа

### Добавление нового Источника заказа

Чтобы добавить новый источник заказа:

* Нажать на кнопку **"+"**, расположенную в нижней части списка Источников заказа.  
* Появится текстовое поле для ввода названия нового источника.  
* Ввести название и нажмите **"Сохранить"**, чтобы добавить источник в справочник.

### Редактирование существующего Источника заказа

Чтобы переименовать существующий источник заказа:

* Нажать на иконку **карандаш** рядом с источником, который необходимо переименовать.  
* Поле названия станет доступным для редактирования.  
* Ввести новое название и нажмите **"Сохранить"**.  
* Должно появится предупреждение:  
  * "Переименование этого источника приведет к его переименованию во всех местах, где он был применен. Вы уверены, что хотите продолжить?".  
* Нажать **"Подтвердить"** для переименования или **"Отмена"**, чтобы прервать действие.

### Удаление Источника заказа

Чтобы удалить существующий источник заказа:

* Нажать на иконку **корзины** рядом с источником, который вы хотите удалить.  
* Появится диалоговое окно с предупреждением:  
  * "Удаление этого источника приведет к его удалению во всех местах, где он был применен. Вы уверены, что хотите продолжить?"  
* Нажать **"Подтвердить"** для удаления или **"Отмена"**, чтобы прервать действие.

Удаление источника убедт является необратимым действием и затронет все области и темы, где этот источник использовался.

## Приложение 2\. Добавление элемент Caption \- Заголовок

В рамках улучшения интерфейса системы Arrivals, добавить поддержку Caption (далее Заголовок) в каждой отображаемое области.  
**Заголовок** представляет собой настраиваемый текстовый элемент, предназначенный для отображения фиксированного текста. **Заголовок** фиксируется по верхнему краю отображаемой области и масштабируется при изменении её ширины.  
Ширина **Заголовка** автоматически адаптируется под текущую ширину области. Высота Заголовка остается фиксированной и не изменяется при изменении размеров области.

**Заголовок** всегда присутствует в структуре области, но его видимость управляется параметром **hidden**.

* **hidden** \= 1\. Заголовок скрыт. Область отображается без заголовка и может расширяться до максимальной высоты, не учитывая границы зоны arrivals.  
* **hidden** \= 0\. Заголовок отображается. При этом основная область автоматически сдвигается вниз, освобождая место для Заголовка.

В настройках области добавляется подраздел "Заголовок" (если Caption скрыт **hidden** \= 1., по умолчанию отображается текст “Новый заголовок”).  
Подраздел "Заголовок" располагается под кнопкой отображения контролла (по умолчанию "Новый контрол").

* В интерфейсе путь к **Заголовку** обозначается как “Тема / Control Area / Заголовок.

При добавлении **Заголовка** (hidden \= 0\) область автоматически сдвигается вниз на фиксированное расстояние (значение параметра отступа для Заголовка), обеспечивая место для Заголовка.  
При скрытии **Заголовка** (hidden \= 0\) область расширяется вверх до предельных размеров, занимая место, освобождаемое **Заголовком**.

### Настройки Заголовка

* **Текст заголовка** Область для ввода статичного текста заголовка. Текст не изменяется независимо от событий в области.  
* **Настройки макета**  
  * **Цвет фона** Возможность выбора цвета фона области Заголовока.  
  * **Прозрачность фона** Настройка уровня прозрачности фона.  
  * **Вертикальные отступы** Фиксированное расстояние **от текста до верхнего и нижнего края** области Заголовка.  
  * **Минимальные расстояния**  
    * **Минимальное расстояние от левого края:** По умолчанию 0\. Актуально,  при выравнивании Left.   
      * Добавить чекбокс “Изменять пропорционально”. По умолчанию \- установлено.  
    * **Минимальное расстояние от правого края:** По умолчанию 0\. Актуально,  при выравнивании Right.  
      * Добавить чекбокс “Изменять пропорционально”. По умолчанию \- установлено.

*При изменении ширины области Заголовка отступы слева “***Минимальное расстояние от левого края”** *и справа “* **Минимальное расстояние от правого края”** *увеличиваются пропорционально ширине области. Например, если ширина области увеличивается в 2 раза, отступы также увеличиваются в 2 раза.*

*Если установлен чекбокс “Изменять пропорционально” и отключен, то пропорциональное увеличение отступа относительно ширина области Заголовка не происходит.*

* **Настройки границы**  
  * **Тип границы** Возможные значения — none, hidden, dotted, dashed, solid, double, groove, ridge, inset, outset.  
  * **Ширина границы** Настройка толщины границы.  
  * **Цвет границы** Выбор цвета границы.  
  * **Скругление углов** Настройка радиуса скругления углов границы.  
* **Настройки шрифта**  
  * **Цвет шрифта** Выбор цвета текста заголовка.  
  * **Размер шрифта** Настройка размера текста.  
  * **Тип шрифта** Возможность изменения типа шрифта.  
  * **Стиль шрифта** Выбор стиля — Normal, Italic, Oblique.  
  * **Начертание шрифта** Настройка начертания текста.  
  * **Выравнивание текста** Возможные варианты — left, center, right, justify.

Необходимо отслеживать изменения ширины области и автоматически корректирует ширину **Заголовка**, сохраняя заданные отступы и выравнивание текста.

При изменении размеров экрана и области, область **Заголовка** масштабируется по ширине, предотвращая искажения и сохраняя пропорциональные отступы.

* При отображении Заголовка **hidden \= 0** основная область сдвигается вниз на фиксированное расстояние, уменьшая свою высоту на размер Заголовка.  
* При скрытии Заголовка **hidden \= 1** основная область расширяется вверх, заполняя освободившееся пространство.

Заголовок реализуется как копия текстового подкласса с добавлением отступов слева и справа для обеспечения гибкости выравнивания.

При создании **Заголовка** фиксируются текущие параметры области, такие как начальная ширина и минимальные отступы от левого и правого краев, равные 0\.

Заголовок размещается вне основной области, чтобы избежать конфликтов при изменении размеров и количества столбцов с контролами.

* Реализовать различные состояния экрана Arrivals (например, режим ожидания, экран заказа) в зависимости от текущих условий.

Реализовать функционал отображения и изменения параметров экранов состояния:

* **Экран заказа** Основной режим, отображается, когда в любой из областей присутствует хотя бы один заказ.  
* **Режим ожидания** Активируется, когда отсутствуют заказы во всех областях.  
* **Пользовательский режим**, Возможность создания и настройки дополнительных режимов.

Необходимо предоставить интерфейс для задания условий, при которых происходит переход между различными состояниями темы. Условия могут основываться на параметрах, таких как количество заказов, количество столбцов и другие метрики, которые не будут добавлены в рамках данной реализации.

Управление режимами отображения темы должно осуществляться через боковую панель Темы. Выпадающий список настройки режима размещается в боковой панели админ-панели настроек темы. По умолчанию выбран режим "Экран заказа" с фиксированным условием отображения.

Условие отображения по умолчанию:

* Экран заказа \- активен, если в любой из областей отображается хотя бы один заказ.  
* Режим ожидания \- активен, если отсутствуют заказы во всех областях.

Для создания нового режима, доступна кнопка  "+" на боковой панели, которая позволяет пользователю панели администрирования.

После нажатия кнопки "+" пользователь вводит название нового режима. При создании нового режима все настройки из текущего выбранного режима (например, "Экран заказа") автоматически копируются в новый режим, чтобы избежать повторения интерфейса и ускорить процесс настройки.

Администратор может переименовывать существующие режимы через боковую панель. Администратор может удалять режимы. При удалении соответствующие настройки темы для этого режима также удаляются.

### Настройка условий отображения режимов

В боковой панели для каждого пользовательского режима доступен редактор условий отображения.

* Интерфейс редактора  
  * **Текстовое поле "Если":** Начало условия.  
  * **Выпадающий список с типами заказов \-** например, доставочный.  
  * **Выпадающий список с статусами заказов \-** администратор выбирает конкретный статус заказа доступный в теме, соответствующий например доставочным заказам (например, "Готовится”).  
  * **Выпадающий список операторов \-** выбор оператора сравнения — "Больше", "Меньше", "Равно", "Больше или равно", "Меньше или равно", “не равно”.  
  * **Поле ввода значения \-** ввод числового значения для сравнения.

Под каждым условием добавить знак “+” и выпадающий список с логическими операторами:

* или  
* и

Таким образом, может одновременно учитываться несколько условий.

Например, "Если "Доставочный", "Готовится", больше 16."

* Данный режим будет отображаться, если количество доставочных заказов в статусе Готовится превышает 16\.  
* Если условие истинно, активируется соответствующий режим. В противном случае работает стандартный режим "Экран заказа".

Система должна отслеживать количество заказов в каждой области в реальном времени. При превышении определенного порога (например, 50 заказов) в заданной области  инициируется изменение состояния режима темы.

#### Алгоритм изменения отображения

* При запуске системы загружаются текущие настройки темы.  
* Проверяются текущие условия для определения активного режима отображения.  
* Читаются события для отслеживания изменений в количестве заказов.  
* При изменении количества заказов инициируется проверку условий.  
* Сравнивается текущее количество заказов с заданными порогами.  
* Определяется, требуется ли переключение на другой режим отображения.  
* Обновляется активный режим отображения на основе результатов проверки условий.  
* Перерисовывается интерфейс с учетом новых настроек.

Для идентификации областей, каждая область получает уникальный идентификатор control\_area\_1, control\_area\_2.

Примеры использования

* **Сценарий 1 \- Низкая нагрузка**

  * Условие \- Количество заказов меньше 50\.

  * Действия Отображается настройка темы для режима “Экран заказа”.

* **Сценарий 2 Высокая нагрузка**

  * Условие **\-** Количество заказов превышает 50\.

  * Действие Отображается настройка темы для пользовательского режима.

* **Сценарий 3 Отсутствие заказов**

  * Условие **\-** Количество заказов равно нулю.

  * Действие  Активируется режим "Ожидания", отображается например только экран компании.

## Приложение 6\. Реализация возможности установки абсолютных координат (x, y) и размеров (ширина, высота) для элементов UI

Необходимо внести изменения в настройки областей, обеспечить точное позиционирование и масштабирование элементов пользовательского интерфейса (UI) в системе Arrivals путем внедрения возможности установки абсолютных координат (x, y) и размеров (ширина, высота) для каждой области ControlArea.

В текущей реализации, все области ControlArea устанавливаются вручную через интерфейс без возможности задания точных координат и размеров через поля конфигурации.

* Необходимо, в панели администрирования, в раздел настроек каждой области ControlArea, добавить параметры для установки абсолютных координат и размеров.

* Отступ от верхнего края рабочей зоны

Обеспечить, что при активном Caption (hidden=0), привязка все равно остается от левого верхнего угла области, а не рабочего пространства.

**Копировать**

* Доступность: Всегда доступна для статусов с существующим дизайном.  
* Функциональность: При нажатии сохраняет текущий дизайн контрола для последующей вставки.

**Вставить**

* Доступность: Доступна только после нажатия кнопки «Копировать» на любом активном статусе с существующим дизайном.  
  * После одной вставки становится неактивной до следующего копирования.  
* Внешний вид при недоступности: Иконка имеет приглушенный синий оттенок и недоступна для нажатия.  
* Функциональность: Вставляет ранее скопированный дизайн контрола в выбранный статус, перезаписывая существующий дизайн.

Вставка дизайна без предварительного копирования невозможна, пользователь не сможет это сделать так как все кнопки “Вставить” будут недоступны для нажатия.

  * Если для выбранного типа статуса остается только один статус с дизайном, кнопка «Удалить» становится недоступной и отображается в приглушенном синем цвете.  
* Функциональность: Удаляет дизайн выбранного статуса и соответствующий контрол.

 Кнопка «Удалить» должна быть недоступна для последнего статуса с существующим дизайном, чтобы предотвратить удаление всех контролов для данного типа статусов.

## Приложение 9\. Изменение контрола

Настройка предназначена для обработки ситуаций, когда изменяемый контрол уже используется в одной или нескольких темах Arrivals. В таких ситуациях при сохранении изменений необходимо предоставить пользователю выбор дальнейших действий посредством изменения текущих кнопок управления на боковой панели: "Сохранить" и "Отмена" заменяются на "Обновить", "Сохранить как" и "Отмена".

При попытке изменить существующий контрол, который уже применяется в одной или нескольких темах Arrivals, система должна выполнять следующие действия:

* Система проверяет, в каких темах используется изменяемый контрол.  
  * Если контрол не используется ни в одной теме, изменения сохраняются автоматически без дополнительных уведомлений.  
  * Если контрол используется в одной или нескольких темах, после внесения изменений и нажатия кнопки "Сохранить", система отображает всплывающее окно с запросом на выбор дальнейших действий.

Обновленные кнопки управления:

* "Обновить" \- Обновляет измененный контрол во всех темах, где он используется.  
* "Сохранить как" \- Создает новый контрол с внесенными изменениями, оставляя оригинальный контрол без изменений.  
* "Отмена" \- Отменяет процесс сохранения изменений, позволяя пользователю вернуться к редактированию.

**Обновить**

Применяет внесенные изменения к существующему контролу во всех темах, где он используется. Сохраняет изменения в базе данных для данного контрола. Обновляет отображение контрола во всех темах, где он используется (автоматически). Отображает всплывающее уведомление: "Контрол успешно обновлен во всех темах."

**Сохранить как**

Создает новый контрол с внесенными изменениями, оставляя оригинальный контрол без изменений. Создает копию измененного контрола с новым уникальным идентификатором. Контрол именуется текстовым исходным значением с приставкой (copy), пользователь при желании, может изменить имя контрола в поле “Имя контрола” самостоятельно. Добавляет новый контрол в систему, доступный для использования в темах. Отображает всплывающее уведомление: "Новый контрол успешно создан."

**Отмена**

Отменяет процесс сохранения изменений, позволяя пользователю вернуться к редактированию контрола. Отменяет все внесенные изменения в текущем сеансе редактирования. Закрывает всплывающее окно. Возвращает пользователя к редактированию контрола без сохранения изменений.

При обнаружении использования контрола в одной или нескольких темах, кнопки "Сохранить" и "Отмена" на боковой панели заменяются на "Обновить", "Сохранить как" и "Отмена".

  * "Обновить": Изменения применяются ко всем темам, кнопки возвращаются к исходным, отображается уведомление.  
  * "Сохранить как": Создается новый контрол с изменениями, кнопки возвращаются к исходным, отображается уведомление.  
  * "Отмена": Изменения не сохраняются, кнопки возвращаются к исходным, отображается уведомление.

  * "Обновить": «Контрол успешно обновлен в темах.»  
  * "Сохранить как": «Новый контрол создан.»

## Приложение 10\. Изменение расположения элементов на основании дочернего элемента

Добавляется функционал выравнивания элементов на основе выбранного основного элемента, с целью эффективной организации отображение информации на табло.

В разделе настроек Темы для Arrivals необходимо настройка добавляется в блок “Выравнивание” добавить новый блок "Выравнивание" в боковую панель для отдельно взятого элемента. Блок "Выравнивание" всегда раскрыт и не имеет возможности свертывания.

Кнопка "Выравнивание": Инициирует режим выравнивания элементов в теме.

* При нажатии на кнопку "Выравнивание" в блоке выравнивания система переходит в режим выбора элементов для выравнивания.

Основной элемент, в настройках которого была нажата кнопка “Выравнивания”, выделяется зеленой сплошной тонкой рамкой. Все элементы, доступные для выбора, получают (остается) красную тонкая рамку. При клике левой кнопкой мыши на элемент с красной рамкой, его рамка меняется на синюю, указывая на признак выбора.

![][image59]

В итоговой конфигурации может быть выбран один основной элемент (зелёная рамка) и несколько дополнительных элементов (синие рамки).

После выбора хотя бы одного дополнительного элемента (один с синей рамкой один с зеленой рамкой)  инициализируются (расдизейблены) кнопки выравнивания.

* Если ни один элемент дополнительный не выбра, кнопки выравнивания задисейблены.

После выбора основного и дополнительных элементов, в блоке "Выравнивание" становятся доступными (расдизейблены) следующие опции:

* По левому краю  
* Центрирование по горизонтали  
* По правому краю  
* По верхнему краю  
* Центрирование по вертикали  
* По нижнему краю

Требования к иконкам и стилистическому исполнению иконок не приводится.

**По левому краю**

Выравнивает левую границу всех выбранных дополнительных элементов по левой границе основного элемента. Считывается значение координаты X левой границы основного элемента. Все выбранные дополнительные элементы изменяют свою координату X так, чтобы их левая граница совпадала с X основного элемента.

![][image60]

**Центрирование по горизонтали**

Центрирует все выбранные дополнительные элементы по горизонтали относительно центральной координаты основного элемента. Вычисляется центральная координата X основного элемента. Все выбранные дополнительные элементы изменяют свою координату X так, чтобы их центральная координата X совпадала с центральной координатой X основного элемента.

![][image61]

**По правому краю**

Выравнивает правую границу всех выбранных дополнительных элементов по правой границе основного элемента. Считывается значение координаты X правой границы основного элемента. Все выбранные дополнительные элементы изменяют свою координату X так, чтобы их правая граница совпадала с X основного элемента.

![][image62]

**По верхнему краю**

Выравнивает верхнюю границу всех выбранных дополнительных элементов по верхней границе основного элемента. Считывается значение координаты Y верхней границы основного элемента. Все выбранные дополнительные элементы изменяют свою координату Y так, чтобы их верхняя граница совпадала с Y основного элемента.

![][image63]

## Приложение 11\. Прилипание объектов

Обеспечить точное позиционирование Элементов, в разделе Темы в рабочей области без необходимости ручного подгонки пикселей пользователем.

* Пользователь перетаскивает один объект (например, квадрат) в сторону другого объекта (например, прямоугольника). Когда объекты находятся вблизи друг друга и пользователь отпускает Левую кнопку Мыши (ЛКМ), квадрат автоматически "притягивается" к прямоугольнику, выравниваясь по соответствующей стороне.

Для активации функции прилипания, на боковую панель “Выравнивание” добавить кнопку активации, по умолчанию включена.

**Механизм работы притягивания**

Реализовать притягивания (snapping) элементов пользовательского интерфейса при их перемещении в рабочей области панели администрирования Темы

При отпускании элемента после перетаскивания могут возникнуть различные сценарии притягивания в зависимости от положения элемента относительно других элементов и границ рабочей области:

* **Горизонтальные границы других элементов \-**  При приближении к левому или правому краю другого элемента, текущий элемент притягивается к соответствующей границе.

![][image73]

* **Вертикальные границы других элементов \-** при приближении к верхнему или нижнему краю другого элемента, текущий элемент притягивается по вертикали.

![][image74]

* **Угловые точки \-** Если элемент приближается к угловой точке другого элемента (например, правый нижний угол), и одновременно находится вблизи соответствующих горизонтальных и вертикальных границ, он притягивается как к границе, так и к углу, обеспечивая выравнивание по обеим осям.

![][image75]

Для реализации функционала притягивания элементов при отпускании левой кнопки мыши необходимо:

* x Координата X верхнего левого угла.  
* y Координата Y верхнего левого угла.  
* width Ширина элемента.  
* height Высота элемента.

Порог притягивания threshold \- Максимальное расстояние в пикселях, при котором происходит притягивание элемента к границе или углу. Настройка должна позволять администратору установить оптимальное значение для удобства пользователей (например, 10 px).

**Алгоритм притягивания при отпускании элемента**

При отпускании элемента запускается процесс проверки на необходимость притягивания. Получаются текущие координаты (x, y) и размеры (width, height) перемещаемого элемента. Получаются координаты и размеры всех статических элементов в рабочей области.

* Если x близко к 0 (левый край) или x \+ width близко к workspace.width (правый край рабочей области) в пределах порога, устанавливается притягивание по горизонтали.  
* Если y близко к 0 (верхний край) или y \+ height близко к workspace.height (нижний край) в пределах порога, устанавливается притягивание по вертикали.

Для каждого статического элемента проверяется, находится ли перемещаемый элемент в пределах порога к его границам. 

* Проверяется расстояние между левыми и правыми границами элементов.  
* Проверяется расстояние между верхними и нижними границами элементов.

Определяются расстояния до угловых точек статических элементов.

* Если одновременно соблюдаются условия близости к горизонтальной и вертикальной границе, устанавливается притягивание к углу.

**Вычисление новых координат элемента**

При горизонтальном притягивании  \- **По левому краю**, X устанавливается равным 0 (если рабочая область) или правому краю статического элемента.

**По правому**, X устанавливается так, чтобы x \+ width совпадало с правым краем рабочей области или левым краем статического элемента.

При вертикальном притягивании \- **По верхнему краю,** Y устанавливается равным 0 или нижнему краю статического элемента.

**По нижнему краю**, Y устанавливается так, чтобы y \+ height совпадало с нижним краем рабочей области или верхним краем статического элемента.

* При совпадении углов и близких расстояний, реализуется одновременное выравнивание по горизонтали и вертикали, например, выравнивание нижних граней.

## Сценарии притягивания

### Горизонтальное притягивание к левому краю рабочей области

Если левый край элемента (x) находится в пределах порога от 0, элемент выравнивается по левому краю: x \= 0\.

Если правый край элемента (x \+ width) находится в пределах порога от workspace.width, элемент выравнивается по правому краю: x \= workspace.width \- width.

Если верхний край элемента (y) находится в пределах порога от 0, элемент выравнивается по верхнему краю: y \= 0\.

Если нижний край элемента (y \+ height) находится в пределах порога от workspace.height, элемент выравнивается по нижнему краю: y \= workspace.height \- height.

### Притягивание к границам других элементов

Горизонтальное притягивание

* Элемент A находится рядом с левым краем элемента B.  
* Элемент A прилипает к правому краю элемента B, устанавливая x \= B.(x \+ width)

Вертикальное притягивание

* Элемент A находится рядом с верхним краем элемента B.  
* Элемент A прилипает к нижнему краю элемента B, устанавливая y \= B.(y \+ height).

Угловое притягивание

* Элемент A находится рядом с правым нижним углом элемента B.  
* Элемент A прилипает к правому краю элемента B и одновременно выравнивается по нижнему краю: x \= B.(x \+ width) и y \= B.(y \+ height).

Приложение 12\. Работа с лицензией

1. ## Режим работы “Нет лицензии”

В отсутствие лицензии плагин функционирует в базовом режиме, выполняя только основные функции DigitalSignage без привязки к дополнительному функционалу Arriivals.

В панели администрирования “Экрана покупателя” настройки связанные с функционалом Arrivals **не** отображается.

Это включает отсутствие вывода \- раздел “Дисплей Arrivals” и все соответствующие подразделы. Темы, Настройка экранов, Контролы \- не доступны пользователю.

2. ## Режим с обычной лицензией

При наличии обычной лицензии активируется базовый набор настроек плагина DigitalSignage, связанный с функционалом Arrivals. В этом режиме для пользователя доступны следующие разделы:

**Дисплей Arrivals** – интерфейс раздела Arrivals, который включает три основные вкладки:

* **Контролы** \- предоставляется режим просмотра контролов без возможности редактирования или удаления. **Темы** \- позволяет выбрать и настроить визуальное оформление экранов.  
* **Настройка экранов** \- предоставляет возможность управления параметрами отображения экранов на кассовых терминалах.

Пользователь имеет возможность просматривать полный список контролов

Должен быть доступен режим предпросмотра с целью пользователю детально ознакомиться с конфигурацией каждого контрола, оценить его визуальное представление.

В режиме обычной лицензии исключены возможности перехода к экрану редактирования контрола и его удаления. API-методы, отвечающие за отображение деталей контрола (GET /api/arrivals/control/{id}), внесение изменений (POST /api/arrivals/control/{id}) и удаление контролов (DELETE /api/arrivals/control/{id}), должны быть исключены из функционала в обычной лицензии, также как и функции (кнопки) удаления и переходу к экрану редактирования.

Все остальные функции панели администрирования DigitalSignage, не относящиеся к разделу Arrivals, остаются доступными.

3. ## Режим с экспертной лицензией

Экспертная лицензия обеспечивает полный доступ ко всему функционалу Arrivals. В режиме эксперта доступны:

**Раздел настройки Arrivals \-**  “Дисплей Arrivals” доступен с ограниченным функционалом. Доступны разделы: 

* Подраздел "Темы" Позволяет выбрать и настроить визуальное оформление экранов.  
* Подраздел "Настройка экранов" Предоставляет возможность управления параметрами отображения экранов на кассовых терминалах.  
* Подраздел "Контролы" Дополнительный блок, предоставляющий функции по настройке контролов \- базовых элементов для конфигурации тем.

Все элементы интерфейса интегрированы и доступны связанные с настройкой экрана электронной очереди Arrivals.

4. ## Техническая реализация

При загрузке плагина система выполняет проверку статуса лицензии. В зависимости от полученного результата формируется соответствующий интерфейс:

* Если лицензия отсутствует – запускается базовый режим без экранов настройки.  
* При наличии обычной лицензии – активируются экран Arrivals, подразделы «Темы» и «Настройка экранов».  
* При наличии экспертной лицензии – дополнительно отображается подраздел «Контролы», предоставляющий расширенные возможности.

Данная спецификация служит ориентиром для разработки и тестирования плагина DigitalSignage с учётом различных сценариев лицензирования.

Приложение 14\. Выделение редактируемого элемента

Обеспечить автоматическое визуальное выделение выбранного (активного) элемента, а также сохранить индивидуальное оформление каждого элемента, заданное посредством его настроек.

1. ## Области элемента

*Каждый элемент на рабочей области Arrivals состоит из нескольких составных частей:*

* Content \- Содержимое, область, в которой отображается основной контент элемента.  
* Padding \- Отступы, внутренние отступы между содержимым и рамкой.  
* Border \- Рамка.  
* Margin \- Внешние отступы.

В настройках темы, рамках любого элемента является частью его настроек, поэтому любое новое визуальное выделение не должно нарушать или заменять оригинальные настройки Border.

На данный момент система предусматривает отдельную кнопку «Выравнивание», при нажатии на которую активный элемент получает зеленую рамку, а все остальные элементы – красную. Такое поведение активируется только после явного нажатия на кнопку «Выравнивание».

2. ## Функциональные требования

При клике левой кнопкой мыши по любому элементу рабочей области (то есть когда элемент становится выбранным и открывается панель настроек для данного элемента) система должна автоматически подсветить выделения элемента, в состояние “активного”.  
Активный элемент должен моментально выделяться **зеленой рамкой**. Этот эффект должен быть реализован посредством наложения отдельного визуального слоя overlay, который имитирует зеленую рамку. Важно, чтобы при этом оригинальная рамка элемента Border, заданная через настройки, оставалась неизменной и отрисовывалась под overlay. Это обозначит, что зеленая рамка будет служить лишь индикатором состояния редактирования, не изменяя параметры внешнего оформления.

1. ## Подробная логика работы

При клике левой кнопкой мыши на элементе происходит событие, которое запускает открытие панели настроек для данного элемента.  
В этот же момент системе передается сигнал, что элемент стал активным. Этот сигнал фиксируется на уровне логики интерфейса, и на его основе начинается процесс визуального выделения.  
При установке состояния «активный элемент» динамически создается специальный слой – overlay, который будет отрисовываться поверх элемента.  
Overlay рамка должна представлять собой визуальную границу с заданными параметрами \- которые идентичны стилистике рамки, проявляющейся через функционал “Выделение”.

Исходная рамка элемента Border должна продолжать отрисовываться согласно установленным параметрам (цвет, толщина, стиль) под overlay рамкой.  
Overlay зеленой рамки накладывается поверх, не затирая и не заменяя визуальные настройки Border.   
Технически это реализуется путем использования слоев с разными z-index, где оригинальный Border имеет более низкий z-index, а overlay – более высокий.  
При изменении размеров или перемещении элемента \- overlay должен автоматически синхронизироваться с изменениями, не влияя на базовую верстку и настройки элемента.

Когда пользователь кликает на другой элемент, overlay с зеленой рамкой начинает отображаться на новом выбранном элементе..

* Состояние текущего активного элемента сбрасывается, и визуальное выделение на текущем элементе исчезает, возвращая элемент к первоначальному виду, где отрисовывается только его оригинальная рамка Border.

  2. ## Интеграция с функционалом кнопки «Выравнивание»

Текущий функционал кнопки «Выравнивание», при котором при нажатии активный элемент выделяется зеленой рамкой, а остальные – красной, сохраняется.  
Автоматическое выделение активного элемента по клику является независимым и выполняется сразу, без ожидания нажатия на кнопку.

* Если пользователь затем активирует режим «Выравнивание», система должна корректно объединять оба состояния, при этом overlay для активного элемента остается зеленым, а остальные элементы будут получать красную рамку.

***Правила применения рамок, при использовании кнопки Выравнивания, должны быть идентичны правилам применения рамки через Overlay, чтобы сохранить оригинальный вид элементов.***
